Our teams use func-style with default options, which means that we stick with function expressions instead of declarations everywhere. We find this better than seeing a mix of declarations (function keyword) and inline arrow functions (no function keyword), which would be the case if we followed typescript-eslint recommendations. See a footnote on func-style in ROADMAP.md, suggesting to use "func-style": ["error", "declaration", { "allowArrowFunctions": true }].

The only problematic case we have is when we do TypeScript-specific function overloads:

// this is the function style we prefer in general
const normalFunc = (x: string) => x + x

// this is what we have to write when there is a need for an overload
function overloadedFunc(x: string): string
function overloadedFunc(x: number): number
function overloadedFunc(x: any): any {
 return x + x
}

Unless we change func-style options, we get the following error in the second case:

Expected a function expression. eslint(func-style)

Function overloading is an advanced TypeScript feature, which we aim to use rarely. Changing hundreds of files because of switching to ["error", "declaration", { "allowArrowFunctions": true }] is not the price we are ready to pay. In addition to a large git diff, the code will become more confusing for JavaScript newcomers – they would need to get exposed into subtle differences between declarations and expressions. Stylistic consistency will suffer as well.

For now, we just use /* eslint-disable func-style */ in files that contain function overloads, but that's not ideal. Here is what we would prefer instead:

Expected Result

 {
   "rules": {
-    "func-style": "error",
+    "@typescript-eslint/func-style": ["error", { allowOverloads: true }],
   }
 }

What do you think of cloning the original rule to support overloads in codebases when function expression are more preferred? This would be a similar case to require-await → @typescript-eslint/require-await. What would be the arguments against implementing such a rule?

Perhaps, there is even no need for an option – allowOverloads can just always be true.

 {
   "rules": {
-    "func-style": "error",
+    "@typescript-eslint/func-style": "error",
   }
 }

lukevella Nov 26, 2019

@phaux Unfortunately this doesn't work if your function parameter is typed to anything other then any.

interface Foo {
  (x: string): string
  (x: number): number
}

const foo: Foo = (x: string | number) => x

Results in the following error:

Type '(x: string | number) => string | number' is not assignable to type 'Foo'.
  Type 'string | number' is not assignable to type 'string'.
    Type 'number' is not assignable to type 'string'. ts(2322)
